# Contributors

`cup-check` riconosce i contributi in modo pubblico, verificabile e proporzionato alla dimensione del progetto. Il repository resta la fonte primaria per cronologia, responsabilità e decisioni.

## Maintainer

Il progetto ha un maintainer operativo:

- Alessandro Saglia (`@ale-saglia`): triage delle issue, revisione e approvazione delle PR, decisioni di roadmap e release.

Al momento non ci sono altri maintainer con permesso di approvazione o merge.

## Contributori Passati

I contributori passati sono tracciati dalla cronologia Git e dalle PR mergiate. Alla data di creazione di questo file risultano contributi nel repository da:

- Alessandro Saglia (`@ale-saglia`)

Nuovi contributori vengono aggiunti a questa sezione dopo il merge del primo contributo rilevante, se desiderano essere citati nominalmente.

## Cosa Conta Come Contributo

Sono contributi riconoscibili:

- codice, test, fixture o documentazione mergiati tramite PR;
- issue riproducibili che portano a una correzione o a una decisione documentata;
- revisione tecnica o amministrativa che cambia una PR, una ADR, la roadmap o una scelta di governance;
- segnalazioni di sicurezza gestite tramite il processo indicato in `SECURITY.md`;
- miglioramenti di accessibilità, usabilità o chiarezza dei messaggi utente.

Per contributi piccoli, come refusi o correzioni di link, il riconoscimento resta nella PR e nella cronologia Git. L'aggiunta a questo file avviene quando il contributo è sostanziale o quando la persona lo richiede.

## Processo Di Riconoscimento

1. Il contributo viene discusso in issue o PR pubblica, salvo segnalazioni di sicurezza.
2. Dopo il merge, il maintainer verifica se aggiornare questa pagina.
3. Se la persona preferisce non comparire, viene rispettata la scelta senza richiedere motivazioni.
4. Se una persona vuole essere citata con nome, ente, handle o solo con handle GitHub, può indicarlo nella PR o in una issue.
5. Le affiliazioni a enti pubblici, aziende o comunità vengono riportate solo se già pubbliche e confermate dalla persona.

Questo processo evita attribuzioni implicite non desiderate e rende chiaro quali contributi hanno inciso sul progetto.

## Nuovi Maintainer

Un contributore può diventare maintainer quando ha dimostrato continuità, cura della qualità e comprensione dei vincoli del progetto:

- web app statica e deployabile su GitHub Pages;
- nessun servizio server-side per il lookup;
- fixture YAML come specifica funzionale;
- separazione tra validazione formale e lookup OpenCUP;
- postura cautelativa sugli esiti utente.

L'aggiunta di un maintainer richiede una PR che aggiorni `CONTRIBUTORS.md` e `CONTRIBUTING.md`, indicando responsabilità operative, ambito di approvazione e canale di contatto.
